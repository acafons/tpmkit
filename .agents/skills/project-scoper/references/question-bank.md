# Scoping Question Bank

Use these prompts selectively. Do not ask every question at once.

## Problem and Purpose

- What problem should this project solve?
- Why is this problem worth solving now?
- What current workflow, process, or limitation motivates this project?
- What would happen if this project were not built?

## Stakeholders

- Who will use the result directly?
- Who will maintain it?
- Who needs to review or approve it?
- Are there security, compliance, platform, or operations stakeholders?

## Objectives

- What outcomes should the project achieve?
- What should be true after the first successful version?
- Which outcomes matter most if tradeoffs are required?

## Scope Boundaries

- What must be included in the first scope?
- What should be explicitly excluded?
- What work is likely but deferred?
- Which responsibilities belong to another project, layer, or team?

## Functional Requirements

- What capabilities must users have?
- What inputs must be accepted?
- What outputs or artifacts must be produced?
- What workflows must be supported end to end?
- What failure cases must be handled?

## Non-Functional Requirements

- What compatibility requirements exist?
- What performance expectations exist?
- What reliability or correctness guarantees matter?
- What security properties are required?
- What portability requirements exist?
- What maintainability expectations should be documented?
- What testability expectations should be documented?

## Constraints and Assumptions

- What technologies, standards, versions, or platforms are fixed?
- What dependencies are required or forbidden?
- What deadlines, policies, or compatibility rules apply?
- What assumptions are being made because details are not known yet?

## Diagrams

- Which users, stakeholders, external systems, infrastructure, and dependencies must appear in the C4 System Context diagram?
- Which runtime, deployable, or logical containers must appear in the C4 Container diagram?
- What are the major relationships and communication paths between diagram elements?
- Who is the intended audience for each mandatory diagram?
- Is Mermaid acceptable for the diagrams, or did the user request another diagram format?

## Success Criteria

- How will the user know the scope is complete?
- How will the team know the first version is successful?
- What observable checks, tests, demos, or documents should exist?

## Risks and Open Questions

- What could make this project harder than expected?
- What could cause incorrect behavior, unsafe behavior, or user confusion?
- What decisions are still unresolved?
- Which questions must be answered before design?
- Which questions can wait until implementation?
