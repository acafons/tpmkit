# Scope: <Name>

> This template describes what each section should contain.
> Do not treat placeholder text as project requirements.
> Replace all instructional text when creating a concrete scope document.

## Overview

Describe, in a short paragraph, what is being scoped and why this scope exists.

## Problem Statement

Describe the problem, pain point, or opportunity this scope addresses.

Avoid describing the solution here unless necessary for context.

## Stakeholders

List the people, roles, or systems affected by this scope.

For each stakeholder, briefly describe their interest or concern.

## Objectives

List the desired outcomes of this scope.

Objectives should describe results, not implementation details.

## In Scope

List what is included in this scope.

Be specific enough that future work can be judged as inside or outside the boundary.

## Out of Scope

List what is explicitly excluded from this scope.

Include deferred work, rejected areas, or responsibilities owned by another scope.

## Functional Requirements

List the behaviors or capabilities the project, feature, or component must provide.

Use requirement IDs when possible, for example:

- FR-001: <Requirement>

Each requirement should be testable or verifiable.

## Non-Functional Requirements

List quality attributes, constraints, or operational expectations.

Examples of categories include:

- compatibility
- performance
- security
- reliability
- maintainability
- portability
- usability
- observability
- testability

Use requirement IDs when possible, for example:

- NFR-001: <Requirement>

Each requirement should be measurable or reviewable where practical.

## Diagrams

Draw the mandatory project-scope diagrams in this section.

Include exactly these C4 diagrams:

- C4 System Context diagram: show the system, users, external systems, infrastructure, and major dependencies.
- C4 Container diagram: show major runtime, deployable, or logical containers and how they communicate.

Use Mermaid code blocks unless the user requests another diagram format.

Do not include a C4 Component diagram in a project scope document. Component diagrams belong in later architecture or design documents after the project scope is accepted.

## Constraints

List fixed limitations, required technologies, standards, policies, deadlines, or compatibility rules.

Constraints are things the project must work within, not goals the project hopes to achieve.

## Assumptions

List things believed to be true while defining this scope.

Assumptions should be revisited if new information appears.

## Success Criteria

List the conditions that must be true for this scope to be considered successful.

Use checkboxes where helpful.

Success criteria should be observable.

## Risks

List uncertainties or possible problems that could affect delivery, quality, security, usability, or maintainability.

Include impact and mitigation when known.

## Open Questions

List unresolved questions that need answers before or during implementation.

Questions should be specific enough that they can be assigned, researched, or decided.

## Deliverables

List the concrete artifacts this scope should produce.

Examples may include documents, code modules, tests, diagrams, decisions, or follow-up scopes.
